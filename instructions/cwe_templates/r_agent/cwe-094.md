### Review Priority

When reviewing CodeQL findings for code injection, focus on whether user-controlled input can actually reach a code evaluation or execution interface in a form that allows arbitrary logic execution. Distinguish between data processing and code execution boundaries.

**Relevant CWEs:**
- CWE-094: Code Injection
- CWE-095: Eval Injection
- CWE-917: Expression Language Injection
- CWE-502: Unsafe Deserialization
- CWE-470: Externally-Controlled Class Selection
**IMPORTANT**: Focus only on the **Relevant CWEs** listed above. Do not include any other CWEs in your analysis or CodeQL queries.
**IMPORTANT**: Prioritize submitting True Positives or Manual Review findings over False Positives.

### Key Validation Criteria

#### 1. Exploitability Assessment
- **Does user input reach the evaluation sink?** Trace backward from the eval/deserialize/lookup call to confirm tainted data is present
- **Is the input interpreted as code?** Distinguish between data parameters (e.g., JSON values) and code parameters (e.g., OGNL expressions, script strings)
- **What is the execution scope?** Can the attacker execute arbitrary code, or only influence a constrained expression?
- **Are there type or syntax restrictions?** Allowlisted classes in deserialization, sandboxed script engines, or restricted expression grammars limit exploitability
- **Is authentication required?** Code injection in admin-only interfaces is lower severity than in public-facing endpoints
- **What expression language features are available?** Check if the EL engine allows method calls, constructor invocations, static method access, or class instantiation — OGNL and SpEL typically allow `Runtime.exec()` via reflection chains; restricted grammars (simple property access only) are much harder to exploit
- **For deserialization: are gadget chain libraries on the classpath?** Check for Commons Collections 3.x/4.x, Commons Beanutils, Spring Beans, Groovy, AspectJ, C3P0, ROME — their presence determines whether ObjectInputStream.readObject() is exploitable for RCE
- **Is the deserialization filter actually restrictive?** ObjectInputFilter or custom `resolveClass()` overrides may use `startsWith` or denylist patterns that are bypassable; verify the filter logic is sound
- **Does the expression engine support method chaining?** Expressions like `"".getClass().forName("java.lang.Runtime").getRuntime().exec("cmd")` can reach arbitrary execution in many EL engines

#### 2. Code Execution Pattern Analysis
- **Direct eval** (CRITICAL): `ScriptEngine.eval(userInput)` or `GroovyShell.evaluate(userInput)` — full code execution
- **Expression language injection** (CRITICAL): OGNL, SpEL, MVEL, JEXL evaluation of user-controlled strings — can typically reach Runtime.exec
- **Unsafe deserialization** (CRITICAL): ObjectInputStream.readObject() on untrusted data — gadget chains enable arbitrary execution
- **JNDI injection** (CRITICAL): InitialContext.lookup(userInput) — enables remote class loading via LDAP/RMI reference
- **Template injection** (HIGH-CRITICAL): user-controlled template content in Freemarker/Velocity/Thymeleaf — can escape sandbox depending on engine
- **Reflection with user input** (MODERATE): Class.forName(userInput) — limited to instantiable classes on the classpath
- **Double evaluation** (SUBTLE): user input stored in context, then evaluated by expression engine in second pass — common OGNL pattern in Struts

#### 3. True Positive Indicators
- User input flows directly to ScriptEngine.eval(), OGNL.getValue(), or expression parser methods
- ObjectInputStream.readObject() processes data from HTTP requests, message queues, or network sockets without type filtering
- XStream.fromXML() or XMLDecoder.readObject() on user-supplied content without security configuration
- JNDI lookup name constructed from request parameters or headers (Log4Shell pattern)
- Struts action attributes using %{...} with user-controlled values (forced OGNL evaluation)
- Class.forName() with user-controlled string, followed by newInstance()
- Groovy, JavaScript, or Python evaluation of user-supplied code strings
- Deserialization with known gadget chain classes on the classpath (Commons Collections, Spring Beans)
- Expression evaluation in error/exception handling paths where error messages contain user input that gets re-evaluated through a string interpolation or expression engine
- Configuration reload mechanisms that re-evaluate dynamic expressions from user-modified configuration sources (database, file, API)
- Double evaluation: user input stored in a data structure (map, model, ValueStack), then the data structure's rendering or serialization triggers expression evaluation on the stored value
- Jackson ObjectMapper with `enableDefaultTyping()` or `@JsonTypeInfo(use=Id.CLASS)` processing user-supplied JSON — enables attacker-controlled class instantiation
- SnakeYAML parsing user-supplied YAML without restricting type tags (`!!`) — `!!javax.script.ScriptEngineManager` and similar tags instantiate arbitrary classes

#### 4. False Positive Indicators
- Expression is evaluated on server-side constants, not user input
- Deserialization uses a type allowlist (ObjectInputFilter, XStream security framework, Jackson without DefaultTyping)
- ScriptEngine is configured with a SecurityManager or sandbox that prevents dangerous operations
- The evaluated expression is constructed entirely from internal logic (e.g., computed field names, not user data)
- JNDI lookup uses a hardcoded name from configuration, not derived from user input
- Code is in test files, build plugins, or CLI utilities not exposed to remote input
- Class loading is restricted to an explicit allowlist of class names
- Deserialization filter uses exact class name matching (not startsWith or package prefix) with a strict allowlist
- Expression language is restricted to simple property access (no method calls, no constructor invocations, no static access) — verify by checking the EL engine configuration
- Jackson's PolymorphicTypeValidator is configured with a strict allowlist of base types (not DefaultTyping.EVERYTHING)

#### 5. Severity Calibration
- **Critical**: Remote unauthenticated code injection with arbitrary execution — OGNL injection, unsandboxed script eval, JNDI with remote codebase, deserialization with gadget chain
- **High**: Authenticated code injection, OR injection constrained to expression language features (may still reach Runtime.exec via OGNL/SpEL method calls)
- **Medium**: Code injection requiring specific classpath conditions (particular gadget chain library present), OR sandboxed execution with escape potential
- **Low**: Theoretical code injection in unreachable code paths, or execution constrained to a safe expression grammar

### Critical Focus Areas

When tracing code:
1. **Identify the execution interface** — is it a script engine, expression language, deserializer, or class loader? Each has different exploitation characteristics
2. **Trace the complete taint chain** — from HTTP/network input through any transformations to the execution call
3. **Check for sandboxing** — script engines may have SecurityManager restrictions; deserialization may have ObjectInputFilter; expression engines may restrict method calls
4. **Assess gadget chain availability** — for deserialization findings, check if known gadget libraries (Commons Collections 3.x, Commons Beanutils, Spring) are on the classpath; also consider Groovy, AspectJ, C3P0, ROME, Hibernate, Vaadin, and Spring AOP
5. **Consider double evaluation** — in frameworks like Struts, user input may be stored in a ValueStack attribute that is later evaluated as OGNL, creating an indirect injection path; also check error handlers and logging that may re-evaluate messages containing user input
6. **Verify the data origin** — is the deserialized data truly from an untrusted source, or is it from an internal service with authenticated channels?
7. **Test the expression capabilities** — construct a mental test: can the expression engine evaluate `"".getClass().forName("java.lang.Runtime")` or `T(java.lang.Runtime).getRuntime().exec("id")`? If method calls are available, the impact is critical regardless of the input pathway
8. **Check deserialization filter soundness** — if an ObjectInputFilter or custom `resolveClass()` is present, verify it uses exact matching or strict allowlisting; denylist-based or startsWith-based filters are often bypassable with alternative gadget chains or class name variations

**IMPORTANT**: When you identify a potential vulnerability, do not stop at the first vulnerable code pattern. Trace the entire call chain to confirm that the vulnerable code is actually reachable with untrusted input, and that there are no mitigating factors that would prevent exploitation.
