### Review Priority

When reviewing CodeQL findings for cross-site scripting, focus on whether user-controlled input actually reaches an HTML or JavaScript output context without adequate encoding. Distinguish between properly escaped output and raw reflection.

**Relevant CWEs:**
- CWE-079: Cross-site Scripting (XSS)
- CWE-080: Basic XSS
- CWE-116: Improper Encoding or Escaping of Output
- CWE-087: Alternate XSS Syntax
**IMPORTANT**: Focus only on the **Relevant CWEs** listed above. Do not include any other CWEs in your analysis or CodeQL queries.
**IMPORTANT**: Prioritize submitting True Positives or Manual Review findings over False Positives.

### Key Validation Criteria

#### 1. Exploitability Assessment
- **Does user input reach the response body?** Trace backward from the output sink to confirm tainted data is present
- **What is the output context?** HTML body, HTML attribute, JavaScript string, URL, CSS — each requires different encoding
- **Is the content type correct?** application/json responses generally cannot trigger XSS in modern browsers; text/html can
- **Is there output encoding?** Check for OWASP ESAPI encoding, Apache Commons StringEscapeUtils, or framework auto-escaping
- **Can the attacker control enough to inject a tag?** Partial control (e.g., only inside an existing attribute value) is harder to exploit than full HTML body injection
- **Does the response set an explicit Content-Type?** Missing Content-Type can enable browser content-type sniffing (treating JSON or plain text as HTML); check for X-Content-Type-Options: nosniff header as a mitigating factor
- **Is the output context inside an HTML comment, script block, or regular element?** Each has different breakout methods and required encoding strategies
- **Is the response served as a file download?** Content-Disposition: attachment generally prevents XSS in modern browsers

#### 2. Output Context Analysis
- **HTML body context** (DANGEROUS if unescaped): user input between tags — attacker can inject <script> or event handlers
- **HTML attribute context** (DANGEROUS if unquoted or unescaped): user input in attribute values — attacker can break out of attribute and inject handlers
- **JavaScript string context** (DANGEROUS): user input in JS literal — attacker can break out of string and execute code
- **URL context** (MODERATE): user input in href/src — attacker can inject javascript: protocol
- **HTTP header context** (LOW for XSS): header injection may enable response splitting but modern servers prevent this
- **JSON API response** (TYPICALLY SAFE): if Content-Type is application/json, browsers will not execute embedded scripts

#### 3. True Positive Indicators
- User input written to response via PrintWriter.write() or similar without HTML encoding
- Error pages that echo request paths, parameters, or exception messages into HTML body
- String concatenation building HTML with request data: `"<div>" + request.getParameter("name") + "</div>"`
- Template output with escaping explicitly disabled
- HTML sanitizer with a permissive policy that allows dangerous tags or attributes (style, svg, math, event handlers)
- Redirect location constructed from user input without URL encoding (can enable header injection)
- JSONP callback parameter reflected without alphanumeric-only validation
- Content-Type not explicitly set on response while writing user-controlled data — browser may sniff as text/html
- User input in `href`, `src`, `action`, or `formaction` attributes without protocol validation (`javascript:`, `data:` schemes)
- User input in `<meta http-equiv="refresh" content="0;url=javascript:...">` patterns
- Multipart responses or server-sent events including user data without encoding in HTML-renderable sections
- Denylists that block `<script>` but miss event handlers (`onfocus`, `onerror`, `autofocus`), SVG elements (`<svg/onload>`), or alternate attribute syntax

#### 4. False Positive Indicators
- Output is properly encoded using OWASP ESAPI, Apache Commons Text, or framework auto-escaping
- Content-Type is application/json or text/plain (not text/html) AND X-Content-Type-Options: nosniff is set
- The output is written to a log file, not an HTTP response
- User input is restricted to a strict allowlist (enum values, numeric IDs)
- The code is in a REST API that never returns HTML content
- Input passes through a robust HTML sanitizer (OWASP Java HTML Sanitizer with default policy) before output
- The response includes Content-Security-Policy headers that prevent inline script execution (defense-in-depth, but should still be flagged as code-level issue)
- Content-Disposition: attachment set on the response — browsers treat as download, not rendering
- User input used only in HTTP response headers (not body) AND modern server-side header injection protections are in place

#### 5. Severity Calibration
- **Critical**: Stored XSS in admin panels or shared content — attacker payload executes for all users
- **High**: Reflected XSS in authentication flows, or stored XSS in user-scoped content
- **Medium**: Reflected XSS requiring user interaction (clicking crafted link), or XSS in non-sensitive pages
- **Low**: Self-XSS (only affects the user who injects), or XSS mitigated by CSP/httpOnly cookies but still present in code

### Critical Focus Areas

When tracing code:
1. **Identify the output sink** — is it response.getWriter(), a template engine, or an API response? This determines the context
2. **Check the Content-Type** — if the response is application/json, XSS is unlikely in modern browsers; but verify X-Content-Type-Options: nosniff is also set
3. **Verify encoding presence** — look for HTML entity encoding (&lt; &gt; &amp; &quot;) applied between the taint source and the output sink
4. **Examine sanitizer configuration** — if an HTML sanitizer is used, check the policy: does it allow style, svg, math, or event handler attributes?
5. **Consider the attack scenario** — can the attacker craft a URL that, when clicked by a victim, executes JavaScript? Or does the payload require stored content?
6. **Check for framework-level protections** — Spring MVC, JSF, and Thymeleaf have auto-escaping by default; findings in these frameworks need evidence of explicit escape bypass
7. **Test URL-context injection** — if user input appears in href, src, or action attributes, check whether `javascript:` and `data:` protocols are blocked; HTML-encoding alone does NOT prevent these URI-based attacks
8. **Verify encoding matches context** — HTML entity encoding is correct for HTML body but insufficient for JavaScript string literals (need JS encoding) or URL parameters (need percent-encoding); mismatched encoding is effectively no encoding

**IMPORTANT**: When you identify a potential vulnerability, do not stop at the first vulnerable code pattern. Trace the entire call chain to confirm that the vulnerable code is actually reachable with untrusted input, and that there are no mitigating factors that would prevent exploitation.
