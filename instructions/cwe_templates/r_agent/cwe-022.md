### Review Priority

When reviewing CodeQL findings for path traversal vulnerabilities, focus on whether user-controlled input can actually escape an intended directory boundary through the identified code path. Distinguish between truly bypassable validation and defense-in-depth mechanisms that are sound.

**Relevant CWEs:**
- CWE-022: Improper Limitation of a Pathname to a Restricted Directory
- CWE-023: Relative Path Traversal
- CWE-036: Absolute Path Traversal
- CWE-073: External Control of File Name or Path
- CWE-706: Use of Incorrectly-Resolved Name or Reference
**IMPORTANT**: Focus only on the **Relevant CWEs** listed above. Do not include any other CWEs in your analysis or CodeQL queries.
**IMPORTANT**: Prioritize submitting True Positives or Manual Review findings over False Positives.

### Key Validation Criteria

#### 1. Exploitability Assessment
- **Can the attacker control enough of the path?** Trace backward from the sink to confirm the tainted portion is sufficient to include traversal sequences (../ or encoded variants)
- **Does canonicalization happen before or after the check?** If the path is normalized (File.getCanonicalPath, Path.normalize, Path.toRealPath) BEFORE the containment check, the check may be sound
- **Is String.startsWith or Path.startsWith used?** String.startsWith("/var/www/public") is bypassable (e.g., "/var/www/publicevil/.."); Path.startsWith is component-aware and generally sound
- **What happens to the path after validation?** If the validated path is subsequently re-interpreted, decoded, or resolved differently, the check may be ineffective
- **Is there a secondary defense?** Container-level security (Jetty SecurityHandler, servlet container path normalization) may mitigate even if application-level checks are weak
- **Can the attacker inject a URL scheme?** If the path flows to a resource loader or utility method that recognizes `file:`, `jar:`, `url:`, or `classpath:` prefixes, injecting a scheme can change the resolution context entirely (e.g., from classpath to absolute filesystem)
- **Are classpath and filesystem resources protected equally?** Check whether both resource handler types (classpath vs external/filesystem) apply the same validation — one may be missing protections present in the other
- **Does the path utility process protocol prefixes?** Methods like `cleanPath()` that split on `:` may interpret `file:../secret` as a protocol-prefixed path, removing the `file:` prefix and processing the remainder differently
- **Are sensitive container directories blocked?** Classpath resource loading without directory filtering can expose `WEB-INF/web.xml`, `META-INF/MANIFEST.MF`, class files, or application configuration

#### 2. Path Validation Soundness Analysis
- **Canonicalize-then-check pattern** (SOUND): path is first resolved to canonical form, then checked against base directory using Path.startsWith — this is correct
- **Check-then-use pattern** (OFTEN VULNERABLE): path is checked as a raw string, then passed to filesystem operations that resolve it differently
- **Decode-after-check pattern** (VULNERABLE): URL decoding or other transformation occurs after the path containment check, allowing encoded traversal sequences to bypass
- **Strip-and-check pattern** (SUSPICIOUS): leading/trailing characters are removed but path traversal sequences in the middle are preserved
- **Partial normalization** (SUSPICIOUS): backslashes are replaced but ../ sequences are not resolved, or vice versa
- **Dual normalization pattern** (SUSPICIOUS): path processed by two different normalization methods sequentially (e.g., `UriPath.canonical()` then `StringUtils.cleanPath()`) — check whether the second normalizer re-introduces attack surface the first one removed (e.g., processing `file:` prefixes, changing separator handling)
- **Scheme-unaware validation** (VULNERABLE): containment check does not account for URL scheme prefixes — path like `file:///etc/passwd` passes a startsWith check for the path component while resolving to an absolute filesystem location
- **Missing deny-list pattern** (VULNERABLE): resource path accepted without filtering sensitive names (`WEB-INF`, `META-INF`, `.class`, `.properties`, `.xml` configuration files)

#### 3. True Positive Indicators
- String-based startsWith() check on a non-canonical path
- URL decoding that occurs after path validation
- Path validation that only strips leading/trailing characters without normalizing internal traversal sequences
- File construction from user input with no containment check at all
- Archive entry names used directly in file creation without traversal checks
- Path.resolve() with user input where the input could be absolute (overriding the base)
- Containment check against a non-slash-terminated base path (prefix collision: "/base" matches "/basement")
- URL scheme prefixes (`file:`, `jar:`, `url:`) accepted in resource paths without validation — allows bridging from classpath resolution to absolute filesystem access
- Resource constructors (e.g., ClassPathResource, resource handler classes) that accept arbitrary path strings without rejecting protocol schemes or residual traversal sequences after cleanup
- Classpath resource handlers that do not restrict access to sensitive container directories (`WEB-INF/`, `META-INF/`)
- Path normalization utilities that split on `:` and process the prefix as a protocol — injecting `file:` changes resolution semantics
- Inconsistent validation across resource types: classpath handler has no path validation while external handler validates paths (or vice versa)

#### 4. False Positive Indicators
- Path is properly canonicalized via File.getCanonicalPath() or Path.toRealPath() before the containment check
- java.nio.file.Path.startsWith() used (component-aware, not string prefix)
- Input is restricted to a known whitelist of filenames (not paths)
- The identified code is in test files, build scripts, or development tooling
- The file path is constructed entirely from server-side configuration with no user input component
- Container or framework provides path normalization before the application code is reached
- Method is a private utility only called with trusted internal arguments (verify all callers)

#### 5. Severity Calibration
- **Critical**: Remote unauthenticated attacker can read arbitrary files (e.g., /etc/passwd, application config, secrets) or write to arbitrary locations
- **High**: Authenticated attacker can escape a per-user directory, OR unauthenticated attacker can read files within a limited scope (sibling directories)
- **Medium**: Path traversal exists but is constrained by secondary defenses (container normalization, chroot, read-only filesystem), or requires specific non-default configuration
- **Low**: Theoretical traversal in utility code that is not reachable from external input in the current codebase, or traversal limited to non-sensitive file types

### Critical Focus Areas

When tracing code:
1. **Follow the path backward** from the file operation to the HTTP input source — confirm the taint chain is complete and unbroken
2. **Identify all normalization and validation steps** in order — draw the sequence: input → decode? → validate? → normalize? → filesystem access
3. **Check ordering** — if decoding or normalization happens AFTER validation, flag as vulnerable
4. **Examine the base path** used in containment checks — is it slash-terminated? Could a prefix collision occur?
5. **Test the bypass mentally** — construct a concrete path like `basedir/../../../etc/passwd` and walk it through each code step to see if it survives validation
6. **Check for secondary paths** — if one validation method is sound but the same file serving logic has an alternate code path that skips it, both must be examined
7. **Test scheme injection** — construct paths like `file:///etc/passwd`, `file:../../../etc/passwd`, or `jar:file:!/path` and trace how they are handled by each normalization and resource-loading method in the chain
8. **Compare handler protections** — if the codebase has both classpath and external/filesystem resource handlers, verify BOTH apply equivalent path validation; a gap in one handler is a separate vulnerability
9. **Check for residual traversal after cleanup** — after all normalization steps, does the final resolved path still contain `../`, `file:`, or other dangerous patterns? Normalizers may leave residuals the containment check does not catch

**IMPORTANT**: When you identify a potential vulnerability, do not stop at the first vulnerable code pattern. Trace the entire call chain to confirm that the vulnerable code is actually reachable with untrusted input, and that there are no mitigating factors that would prevent exploitation.
