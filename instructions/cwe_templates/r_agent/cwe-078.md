### Review Priority

When reviewing CodeQL findings for OS command injection, focus on whether user-controlled input can actually influence the command string or arguments that reach an OS execution interface. Distinguish between parameterized execution (safe) and shell-interpreted execution (dangerous).

**Relevant CWEs:**
- CWE-078: OS Command Injection
- CWE-077: Command Injection
- CWE-088: Argument Delimiter Injection
**IMPORTANT**: Focus only on the **Relevant CWEs** listed above. Do not include any other CWEs in your analysis or CodeQL queries.
**IMPORTANT**: Prioritize submitting True Positives or Manual Review findings over False Positives.

### Key Validation Criteria

#### 1. Exploitability Assessment
- **Does user input reach the command string?** Trace backward from exec/ProcessBuilder to confirm tainted data flows into the command
- **Is it shell-interpreted?** Runtime.exec(String) passes through a shell; Runtime.exec(String[]) does not — the single-string form is far more dangerous
- **Can the attacker inject metacharacters?** Semicolons, pipes, backticks, $() — if these are not stripped or escaped, injection is possible
- **Is there input validation?** Allowlists are strong; denylists and regex filters are often bypassable
- **What privileges does the process run with?** Command injection as root/SYSTEM is critical; as a sandboxed user is lower severity
- **Does the target command support dangerous flags?** Many Unix tools have flags that enable code execution (e.g., `find -exec`, `tar --checkpoint-action=exec=`, `git --upload-pack`, `rsync -e`, `curl -o` for arbitrary file write) — even array-form execution can be exploitable if the user controls flag positions
- **Are newlines, carriage returns, or null bytes filtered?** These can terminate commands in shell contexts or split a single command into multiple operations
- **Is the working directory attacker-influenceable?** Some tools resolve relative paths from cwd, enabling path-based attacks

#### 2. Command Execution Pattern Analysis
- **Direct shell execution** (DANGEROUS): `Runtime.exec("cmd /c " + userInput)` or `Runtime.exec("/bin/sh -c " + userInput)` — full shell injection
- **Array-form execution** (SAFER): `Runtime.exec(new String[]{"/usr/bin/tool", userInput})` — no shell metacharacter interpretation, but argument injection may still be possible
- **ProcessBuilder with list** (SAFER): `new ProcessBuilder(Arrays.asList("tool", userInput))` — similar to array form
- **Indirect via control protocol** (CONTEXT-DEPENDENT): shutdown commands, JMX operations, management interfaces — exploitability depends on authentication and access control
- **Script engine evaluation** (DANGEROUS): `ScriptEngine.eval(userInput)` — equivalent to code injection even without OS command execution

#### 3. True Positive Indicators
- User input concatenated into a command string passed to Runtime.exec(String)
- ProcessBuilder constructed with user input inserted into the command list without sanitization
- Shell metacharacters (;|`$) not filtered from user input before command construction
- SSH command execution with user-controlled command strings
- Management/control interfaces (shutdown, restart) accepting unauthenticated input
- Deserialization triggering command execution (e.g., Commons Collections gadget chains)
- User input used as arguments to tools that support execution flags (e.g., `git` with `--upload-pack`, `tar` with `--checkpoint-action=exec=`, `find -exec {}`)
- Missing `--` option terminator before user-controlled positional arguments in array-form execution — allows flag injection even without shell metacharacters
- Filenames controlled by the user that start with `-` and are passed as command arguments without path prefix (e.g., `./-file` vs `-file`)
- Newline characters (`\n`) in user input that can split a single logical command into multiple commands on shells that interpret line breaks

#### 4. False Positive Indicators
- Command is constructed entirely from server-side constants with no user input component
- User input is restricted to a strict allowlist (enum values, predefined command names)
- Array-form execution where user input is confined to a single argument that cannot be interpreted as a flag
- The execution path requires local access or admin authentication that limits the attack surface
- The code is in test files, CLI tools, or admin-only utilities not exposed to remote input
- ProcessBuilder command list is fully hardcoded

#### 5. Severity Calibration
- **Critical**: Remote unauthenticated command injection with arbitrary command execution capability
- **High**: Authenticated command injection, OR unauthenticated but limited to specific commands (e.g., can only control arguments to a fixed command)
- **Medium**: Command injection requires chaining with another vulnerability, OR is limited to argument injection in array-form execution
- **Low**: Theoretical injection in code not reachable from external input, or execution is sandboxed/containerized with minimal privileges

### Critical Focus Areas

When tracing code:
1. **Identify the execution method** — is it single-string (shell) or array-form (no shell)? This determines whether metacharacters are interpreted
2. **Trace the taint chain completely** — from HTTP input through any transformations to the exec call
3. **Check for sanitization** — is user input validated, escaped, or restricted before reaching the command? Allowlists are strong; denylists are weak
4. **Assess the execution context** — what user/permissions does the process run as? Is it inside a container or sandbox?
5. **Consider indirect execution** — even if the direct exec call uses constants, does user input control which branch executes (e.g., choosing between "start" and "shutdown")?
6. **Check argument position** — even in array-form execution, user input at flag positions is dangerous; verify whether a `--` option terminator separates flags from user-controlled arguments
7. **Identify the invoked tool** — research whether it has built-in execution capabilities (e.g., `git`, `tar`, `find`, `rsync`, `vim`, `awk`) that an attacker could leverage through argument injection

**IMPORTANT**: When you identify a potential vulnerability, do not stop at the first vulnerable code pattern. Trace the entire call chain to confirm that the vulnerable code is actually reachable with untrusted input, and that there are no mitigating factors that would prevent exploitation.
