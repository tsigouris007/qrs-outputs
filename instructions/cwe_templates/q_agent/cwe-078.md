### Priority Areas

Focus CodeQL analysis on system command execution patterns, process creation, and shell invocation. The primary concern involves user-controlled input reaching operating system command execution interfaces without adequate sanitization or parameterization.

**Relevant CWEs:**
- CWE-078: Improper Neutralization of Special Elements used in an OS Command ('OS Command Injection')
- CWE-077: Improper Neutralization of Special Elements used in a Command ('Command Injection')
- CWE-088: Improper Neutralization of Argument Delimiters in a Command
**IMPORTANT**: Focus only on the **Relevant CWEs** listed above. Do not include any other CWEs in your analysis or CodeQL queries.

### Analysis Strategy

#### 1. Direct Command Execution
Examine all paths where strings are passed to OS command execution:
- Runtime.getRuntime().exec() with string arguments (shell-interpreted)
- ProcessBuilder construction with user-influenced command components
- Channels and shell sessions (JSch, Apache MINA SSHD, Expect4j)
- Script engine evaluation (ScriptEngine.eval, GroovyShell, BeanShell)
- JMX MBean invocations that execute system commands
- JNDI lookups that may trigger remote code loading

#### 2. Shell Metacharacter Injection
Investigate command string construction patterns:
- String concatenation building shell commands from user input
- Semicolons (;), pipes (|), backticks (`), $() subshells, && and || operators in user-controlled portions
- Unquoted or improperly quoted arguments in shell command strings
- String.format or MessageFormat building command lines with user data
- Environment variable injection through user-controlled values
- Newline injection (`\n`, `\r\n`) in command strings — can terminate the current command and start a new one on some shells
- Whitespace and IFS manipulation — spaces, tabs, and custom IFS values affecting argument splitting
- Heredoc (`<<`), process substitution (`<()`), and brace expansion (`{a,b}`) patterns in bash-interpreted commands
- Glob/wildcard injection — user input containing `*`, `?`, or `[...]` that the shell expands to filenames

#### 3. Indirect Command Execution
Analyze patterns where commands are executed through intermediary layers:
- Process control commands (shutdown, restart, stop) accepting user parameters
- Task schedulers and job runners executing user-specified commands
- Plugin or extension loading mechanisms that accept user-controlled class/module names
- SSH command execution with user-controlled command strings
- Database stored procedures that execute OS commands (xp_cmdshell patterns)
- Dynamic language runtime invocations (e.g., invoking Python/Ruby/Node with user-controlled script paths or `-c` arguments)
- Cron expressions or scheduled task definitions where user controls the command field
- Container orchestration commands (kubectl exec, docker exec/run) with user-influenced parameters
- Mail system commands (sendmail, postfix) with user-controlled recipient or header values that reach shell interpretation

#### 4. Argument Injection
Examine cases where user input becomes command arguments:
- git, docker, curl, wget invocations with user-controlled URLs or parameters
- Build tool invocations (mvn, gradle) with user-influenced arguments
- Flag injection: user input prepended with -- or - to inject additional flags
- File path arguments that could contain shell metacharacters
- Tools with built-in execution flags: `git` with `--upload-pack`/`--receive-pack`, `tar` with `--checkpoint-action=exec=CMD`, `find` with `-exec`, `rsync` with `-e` or `--rsh`, `curl` with `-o` (arbitrary file write)
- Option terminator bypass: missing `--` separator before user-controlled positional arguments allows flag injection
- Filename-as-argument attacks: filenames starting with `-` treated as flags by Unix tools (e.g., `rm -rf` with a file named `--no-preserve-root`)
- URL arguments containing embedded credentials or redirects that a tool follows, reaching internal services (SSRF via command argument)

#### 5. Taint Flow from Input to Execution
Trace data flow from untrusted sources to command execution:
- HTTP request parameters, headers, or body data flowing to exec/ProcessBuilder
- Message queue payloads reaching command execution
- Configuration values from untrusted sources (user-editable config, database entries) used in commands
- Deserialized objects that trigger command execution in their lifecycle methods

### Critical Focus Areas

Write CodeQL queries targeting:
1. **Taint paths from remote input to command execution** — trace from RemoteFlowSource to Runtime.exec(), ProcessBuilder, or ProcessBuilder.command()
2. **String concatenation in command construction** — find string concatenation or format operations that build command strings with tainted components
3. **Unparameterized command execution** — find Runtime.exec(String) (single-string form, shell-interpreted) vs Runtime.exec(String[]) (array form, no shell)
4. **Indirect execution via control commands** — find methods named shutdown, stop, restart, execute, or run that accept string parameters from external input
5. **Argument injection in tool invocations** — find ProcessBuilder or exec calls where user input is inserted into argument arrays without validation
6. **Dangerous tool flags** — find invocations of `git`, `tar`, `find`, `rsync`, `curl`, `wget`, `ssh`, `scp` where user input can reach flag positions; check for missing `--` option terminator before user-controlled arguments
7. **Newline and control character injection** — find command string construction where user input is not filtered for `\n`, `\r`, `\0`, or other control characters that may terminate or split commands
