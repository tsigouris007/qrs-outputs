### Priority Areas

Focus CodeQL analysis on dynamic code evaluation, expression language injection, unsafe deserialization, and runtime class loading. The primary concern involves user-controlled input reaching code execution interfaces that interpret or compile the input as executable logic.

**Relevant CWEs:**
- CWE-094: Improper Control of Generation of Code ('Code Injection')
- CWE-095: Improper Neutralization of Directives in Dynamically Evaluated Code ('Eval Injection')
- CWE-917: Improper Neutralization of Special Elements used in an Expression Language Statement
- CWE-502: Deserialization of Untrusted Data
- CWE-470: Use of Externally-Controlled Input to Select Classes or Code
**IMPORTANT**: Focus only on the **Relevant CWEs** listed above. Do not include any other CWEs in your analysis or CodeQL queries.

### Analysis Strategy

#### 1. Expression Language and Template Injection
Examine paths where user input reaches expression evaluation engines:
- OGNL evaluation (Struts ValueStack, ActionContext) with user-controlled expressions
- Spring Expression Language (SpEL) evaluation with user input
- MVEL, JEXL, or other expression language interpreters processing external data
- EL injection in JSP/JSF via unescaped expression evaluation
- Template engines (Freemarker, Velocity, Thymeleaf, Pebble) with user-controlled template strings
- Server-Side Template Injection (SSTI) via user-controlled template content
- Property resolution chains in EL/OGNL: expressions like `${pageContext.servletContext}`, `${applicationScope}`, or `${T(java.lang.Runtime)}` reaching sensitive runtime objects
- Method invocation chains in SpEL/OGNL: `.getClass().forName("java.lang.Runtime").getMethod("exec",...)` enabling arbitrary command execution from expression context
- Nested or double evaluation: outer expression evaluates a string that itself contains expression syntax with user input — common pattern in Struts `%{...}` forced evaluation
- Expression evaluation in error/exception handling paths where error messages containing user input are processed through an expression engine a second time

#### 2. Script Engine Evaluation
Investigate Java scripting API usage:
- javax.script.ScriptEngine.eval() with user-influenced script content
- Groovy shell or script evaluation (GroovyShell.evaluate, GroovyClassLoader.parseClass)
- JavaScript engine (Nashorn, GraalJS) executing user-supplied scripts
- BeanShell interpretation of user-controlled input
- Python/Ruby interpreters invoked from Java with user data

#### 3. Unsafe Deserialization
Analyze deserialization patterns:
- ObjectInputStream.readObject() processing untrusted data streams
- XMLDecoder reading user-supplied XML
- XStream deserialization without type allowlists
- Jackson or Gson polymorphic deserialization with attacker-controlled type identifiers (`@JsonTypeInfo`, `enableDefaultTyping()`, `@type` fields)
- YAML deserialization (SnakeYAML) with arbitrary object instantiation via `!!` type tags
- Java RMI, JMX, or T3/IIOP protocols accepting serialized objects from untrusted sources
- Serialization gadget chains (Commons Collections, Commons Beanutils, Spring beans)
- Custom `readObject()`, `readResolve()`, or `readExternal()` implementations that perform dangerous operations (file I/O, command execution, JNDI lookups) during deserialization
- Proxy-based gadget chains: `java.lang.reflect.Proxy` with `InvocationHandler` implementations that trigger dangerous operations on method invocation
- Second-order deserialization: deserialized object later triggers another deserialization in a different context (e.g., a cached serialized object re-deserialized with fewer restrictions)
- Partial deserialization filters: `ObjectInputStream` with `resolveClass()` override — verify the override is actually restrictive and not bypassable (e.g., only checking class names with `startsWith` rather than exact matching)

#### 4. Dynamic Class Loading and Reflection
Examine runtime code loading mechanisms:
- Class.forName() or ClassLoader.loadClass() with user-controlled class names
- Reflection-based method invocation (Method.invoke) where method name or target comes from user input
- Service loader patterns with user-influenced service provider names
- URLClassLoader with attacker-controlled URLs
- JNDI lookup (InitialContext.lookup) with user-controlled names — can trigger remote class loading
- ServiceLoader with user-influenced META-INF/services content — custom service provider loading
- MethodHandle and VarHandle based invocations with user-controlled lookup targets
- Thread context classloader manipulation followed by class loading — attacker may influence the classloader to load malicious classes
- Spring Bean definition injection: user input influencing bean class names, factory methods, or property values in IoC container configuration

#### 5. Taint Flow from Input to Code Execution
Trace data flow from untrusted sources to evaluation sinks:
- HTTP request parameters flowing to expression language evaluators
- Message queue payloads reaching script engines
- Configuration values from untrusted sources used in reflection or class loading
- Protocol messages (MQTT, STOMP, OpenWire) with fields that trigger code execution during parsing
- RPC frameworks (Dubbo, gRPC) where method names or payloads influence code dispatch

### Critical Focus Areas

Write CodeQL queries targeting:
1. **Taint paths from remote input to expression evaluation** — trace from RemoteFlowSource to ScriptEngine.eval(), OGNL.getValue(), SpelExpressionParser.parseExpression(), or similar evaluation sinks
2. **Deserialization from untrusted sources** — find ObjectInputStream.readObject(), XMLDecoder.readObject(), XStream.fromXML() where the input stream originates from network data or user-uploaded content
3. **JNDI injection** — find InitialContext.lookup() or similar where the JNDI name is constructed from user input (enables remote class loading via LDAP/RMI)
4. **Dynamic class instantiation from input** — find Class.forName(userInput), ClassLoader.loadClass(userInput), or Constructor.newInstance() where the class name traces to external data
5. **Expression language double evaluation** — find patterns where user input is first stored in a context variable and then evaluated by an expression engine in a second pass (common in Struts OGNL injection)
6. **Gadget chain reachability** — for deserialization sinks, check whether known gadget libraries are on the classpath (Commons Collections 3.x/4.x, Commons Beanutils, Spring Beans, Groovy, AspectJ, C3P0, ROME) to assess real exploitability
7. **Polymorphic deserialization configuration** — find Jackson ObjectMapper with `enableDefaultTyping()`, `@JsonTypeInfo(use=Id.CLASS)`, or XStream without `allowTypes()` / `setupDefaultSecurity()` configuration — these enable attacker-controlled type instantiation
8. **Expression evaluation in error paths** — find exception handlers, error formatters, or logging frameworks where error messages containing user input pass through expression evaluators (e.g., string interpolation engines, Log4j message lookups)
