### Priority Areas

Focus CodeQL analysis on file system access patterns, path construction, and input validation logic. The primary concern involves user-controlled input reaching file system operations without adequate path canonicalization or containment checks.

**Relevant CWEs:**
- CWE-022: Improper Limitation of a Pathname to a Restricted Directory ('Path Traversal')
- CWE-023: Relative Path Traversal
- CWE-036: Absolute Path Traversal
- CWE-073: External Control of File Name or Path
- CWE-706: Use of Incorrectly-Resolved Name or Reference
**IMPORTANT**: Focus only on the **Relevant CWEs** listed above. Do not include any other CWEs in your analysis or CodeQL queries.

### Analysis Strategy

#### 1. Path Construction and Resolution
Examine how file paths are built from external input:
- String concatenation used to build file paths from request parameters, headers, or URI segments
- Path joining operations (File constructor, Paths.get, Path.resolve) with user-influenced components
- Missing or incomplete path normalization before directory containment checks
- Differences between logical path validation and actual filesystem resolution
- URL scheme/protocol prefixes injected into path strings (`file:`, `jar:`, `url:`, `classpath:`, `jndi:`, `http:`, `ftp:`) — these can change how the path is resolved by resource loaders or utility methods
- Path utility methods (e.g., `cleanPath()`, `normalize()`) that split on `:` and treat the left side as a protocol prefix, altering the resolved path
- Double canonicalization: path processed by multiple normalization methods with different semantics (e.g., one resolves `..` sequences, another processes URL schemes) — the interaction between normalizers can introduce bypasses

#### 2. Directory Containment Checks
Investigate boundary enforcement mechanisms:
- String prefix checks (startsWith) used for path containment — these are bypassable
- Path canonicalization timing: is the path normalized BEFORE or AFTER the containment check?
- Handling of symbolic links, junction points, and filesystem aliases
- Comparison between string-based validation and proper Path-based containment (Path.startsWith vs String.startsWith)
- Trailing slash presence or absence affecting prefix matching

#### 3. Encoding and Normalization Bypass
Analyze input transformations that occur before or after validation:
- URL decoding (URLDecoder.decode, percent-encoded sequences) timing relative to path validation
- Double encoding attacks (%252e%252e decodes to %2e%2e then to ..)
- Backslash-to-forward-slash normalization on Windows vs Unix
- Null byte injection in path strings (%00 or \0 truncating path validation)
- Unicode normalization of path separators (fullwidth slash U+FF0F, division slash U+2215)
- Path segment sequences: dot-dot (../), dot-dot-backslash (..\..), and variations
- Protocol/scheme prefix injection: `file:///etc/passwd` or `file:../../../etc/passwd` used where only relative paths are expected — ClassLoader and resource utility methods may resolve `file:` prefixes to absolute filesystem paths
- Semicolon path parameters: `/resource;param=value/..` where the semicolon portion is stripped at different stages by different components (servlet container vs application)
- Fragment identifiers and query strings in path components that may be handled inconsistently across normalization layers
- Case sensitivity mismatches: validation on case-normalized path but filesystem resolution is case-sensitive (or vice versa)

#### 4. Static File Serving and Resource Loading
Examine resource retrieval mechanisms that serve files from a configured base directory:
- Static resource handlers and their path resolution logic
- Classpath resource loading with user-influenced paths — ClassLoader.getResource() and getResourceAsStream() resolve paths differently than filesystem File operations
- External (filesystem) resource handlers that may use different validation than classpath handlers — check both paths
- Archive extraction (ZIP/TAR) entry name validation — zip slip patterns
- Temporary file creation with user-influenced names
- Sensitive directory access via classpath resources: `WEB-INF/`, `META-INF/`, configuration files, or class files reachable through classpath resource loading
- Resource handlers that lack input validation at the resource constructor level — validation may only exist in the handler but not in the resource class itself
- URL scheme injection in classpath resources: user input containing `file:` prefix can bridge from classpath resolution to absolute filesystem access when passed to ClassLoader or resource utility methods
- Inconsistent protection between classpath and external resource handlers — one may have strict validation while the other lacks it entirely

#### 5. Taint Flow from HTTP Input to Filesystem
Trace data flow from untrusted sources to file operations:
- Request URI, path info, query parameters flowing to File/Path constructors
- Route parameters and wildcard/splat captures used in file operations
- Multipart upload filename handling
- Header values (Content-Disposition, X-Forwarded-* ) reaching path operations

### Critical Focus Areas

Write CodeQL queries targeting:
1. **Taint paths from HTTP input to file system access** — trace from RemoteFlowSource or servlet request methods to File, Path, FileInputStream, FileOutputStream, ClassLoader.getResource, and similar sinks
2. **Inadequate path validation patterns** — find String.startsWith() checks on paths, especially when the path is not canonicalized first (missing Path.normalize(), File.getCanonicalPath(), or toRealPath())
3. **Decode-after-validate patterns** — locate URLDecoder.decode() calls that occur after path validation or containment checks have already been performed
4. **Direct file construction from input** — find new File(userInput) or Paths.get(userInput) where userInput traces back to HTTP request data
5. **Archive extraction without entry validation** — find ZipEntry.getName() flowing to File constructors without checking for path traversal sequences
6. **Protocol scheme injection in resource paths** — find ClassPathResource, ClassLoader.getResource(), or similar resource-loading APIs where user input can include `file:`, `jar:`, `url:`, or other protocol prefixes that switch the resource resolution context from classpath to filesystem
7. **Missing sensitive directory filtering** — find resource paths that do not reject access to `WEB-INF/`, `META-INF/`, or other sensitive container directories
8. **Asymmetric handler validation** — find codebases with multiple resource handlers (classpath vs external/filesystem) where protections differ, and one handler lacks validation present in the other
