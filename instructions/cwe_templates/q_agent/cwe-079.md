### Priority Areas

Focus CodeQL analysis on output encoding, HTML generation, and user input reflection patterns. The primary concern involves user-controlled input being included in HTTP responses, HTML documents, or JavaScript contexts without adequate output encoding or sanitization.

**Relevant CWEs:**
- CWE-079: Improper Neutralization of Input During Web Page Generation ('Cross-site Scripting')
- CWE-080: Improper Neutralization of Script-Related HTML Tags in a Web Page (Basic XSS)
- CWE-116: Improper Encoding or Escaping of Output
- CWE-087: Improper Neutralization of Alternate XSS Syntax
**IMPORTANT**: Focus only on the **Relevant CWEs** listed above. Do not include any other CWEs in your analysis or CodeQL queries.

### Analysis Strategy

#### 1. Reflected XSS — Input to Response
Examine paths where HTTP request data is written back to responses:
- Request parameters, headers, or URI components written to HttpServletResponse
- Error messages that include request data (404 pages echoing the requested path, exception messages containing input)
- Redirect URLs constructed from user input without encoding
- Search results or form field pre-population with unescaped user data
- API responses that return user input in HTML or JavaScript content types
- Response headers written with user input reflected in auto-generated error pages (e.g., `Host` header injection causing XSS in default error responses)
- WebSocket message handlers that echo user messages back to clients without encoding
- Server-Sent Events (SSE) streams that include unencoded user data in event payloads
- Content-Type not explicitly set on response — browser may sniff content as text/html, treating non-HTML as executable

#### 2. HTML and Template Output
Investigate HTML generation patterns:
- String concatenation building HTML with user data (most dangerous)
- Template engines with auto-escaping disabled or bypassed (raw/unescaped output directives)
- JSP expression language (EL) without output encoding: ${userInput}
- JSTL tags without escapeXml: <c:out value="${param.input}" escapeXml="false"/>
- Servlet PrintWriter.write() or println() with HTML content containing user data
- Response.getWriter() or Response.getOutputStream() writing unescaped user input

#### 3. HTML Sanitization Bypass
Analyze sanitization and filtering mechanisms:
- Custom HTML sanitizers that may be incomplete (missing event handlers, SVG/MathML elements, data URIs)
- Allowlist-based sanitizers where the allowed tags or attributes permit script execution (style with expression(), onload, onerror)
- Sanitizer bypass via encoding (HTML entities, Unicode, null bytes)
- DOM-based sanitization that can be bypassed via mutation XSS (mXSS) — sanitized HTML re-parsed by the browser creates executable content from apparently safe markup
- Libraries like AntiSamy or OWASP Java HTML Sanitizer with misconfigured policies
- Namespace confusion: SVG (`<svg>`) and MathML (`<math>`) elements in HTML5 parsing contexts that bypass tag-based sanitization (e.g., `<svg/onload=alert(1)>`)
- CSS-based attacks: `expression()` in older IE, `-moz-binding`, `behavior` properties, and `@import url(javascript:...)` in style contexts
- Data URI injection in `href` or `src` attributes: `data:text/html,<script>alert(1)</script>` — check if sanitizer strips `data:` protocol in URL attributes
- Denylists that miss less common event handlers (`onfocus`, `onmouseover`, `ontoggle`, `onauxclick`, `onanimationend`) or SVG-specific handlers (`onbegin`, `onrepeat`)

#### 4. JavaScript Context Injection
Examine JavaScript output contexts:
- User data embedded in JavaScript string literals without JS encoding
- JSON responses served with HTML content type
- JSONP endpoints reflecting callback parameters
- Inline script blocks with interpolated server-side values
- Event handler attributes (onclick, onerror) with user-controlled values
- Template literal injection: user data in JavaScript template strings (backtick strings) can inject `${expression}` syntax for code execution
- DOM clobbering: user-controlled HTML `id` or `name` attributes that override JavaScript global variables or DOM API properties
- Prototype pollution via user input that modifies `Object.prototype` through merge/extend utilities, indirectly enabling XSS through polluted properties
- PostMessage handlers that evaluate or render received message data without origin validation

#### 5. Taint Flow from Input to HTML Output
Trace data flow from untrusted sources to output sinks:
- HttpServletRequest.getParameter() → response.getWriter().write()
- Request URI or path info → error page content
- Database-stored user input → HTML rendering (stored XSS)
- HTTP headers (Referer, User-Agent) → log viewers or admin pages

### Critical Focus Areas

Write CodeQL queries targeting:
1. **Taint paths from HTTP input to response output** — trace from RemoteFlowSource to PrintWriter.write(), ServletResponse.getWriter(), or response output stream operations
2. **Unencoded output in error handlers** — find catch blocks or error pages that include exception messages or request data in response output
3. **HTML string construction** — find string concatenation that builds HTML fragments with tainted data, especially in servlet doGet/doPost methods
4. **Sanitizer bypass patterns** — find HTML sanitization calls (AntiSamy, OWASP Sanitizer) and check if the sanitized output can still contain executable content
5. **Template injection without escaping** — find template rendering operations where auto-escaping is explicitly disabled or user input reaches raw/unescaped output directives
6. **Missing Content-Type enforcement** — find response paths that write user data without explicitly setting Content-Type to a safe value — missing headers allow browser content-type sniffing that may interpret responses as HTML
7. **Data URI and JavaScript URI sinks** — find user input flowing into `href`, `src`, `action`, or `formaction` attributes without protocol validation — `javascript:` and `data:text/html,` schemes enable XSS even when HTML tags are escaped
8. **DOM clobbering vectors** — find patterns where user-controlled HTML attributes (`id`, `name`) can override global variables or DOM API properties used in downstream JavaScript
